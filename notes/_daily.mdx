05/20/26:

- Great read from the A11Y collective, [aria-label vs title](https://www.a11y-collective.com/blog/aria-label-vs-title/). I need to rethink my usage of the title attribute, especially for icon buttons! 
- `title` is a native HTML attribute that gives a tooltip when you hover over an element. I had been under the impression that title would be sufficient for providing screen readers with additional context when interacting with elements that were purely visual (e.g. icon buttons). I.e. my thinking was, if you interact with the element, then the screen reader would announce the content of the `title` attribute, giving you an idea of what will happen if clicked. Buuuut this is incorrect, because:
  - the content of the title attribute is often ignored or read inconsistently
  - **it’s dependent on mouse hover or keyboard focus.** It won’t be accessible on mobile devices or touchscreens, where there’s no concept of hovering.
- **TLDR:** the `title` attribute is for displaying tooltips to **sighted users**. `aria-label` should be used if you want certainty that the content will be announced by a screen reader. BUT! see [When to use aria-label](https://www.a11y-collective.com/blog/aria-label-vs-title/#h-when-to-use-aria-label), because aria-label sits at the bottom of the priority list, we should be prioritizing native text, elements (labels), and `aria-labelledby`!

04/27/26:
- Explored the various modules of [La Suite numérique](https://github.com/suitenumerique/), the French government's "digital workspace for online collaboration and teamwork." While looking through the [meet module](https://github.com/suitenumerique/meet), they reference a project called [LiveKit](https://livekit.com/) that powers the video conferencing features. And LiveKit looks cool! (Aside from all the agentic AI shit they've been adding 🙃). I'm like halfway through [their blog post about the v1 release](https://livekit.com/blog/livekit-one-dot-zero) (which was like 4 years ago), but they're talking about how they do adaptive streaming with WebRTC -- since adaptive streaming is a _subscriber-side_ feature, the LiveKit client SDK handles tracking which video elements are visible and their sizes (e.g. 20 person call will have a bunch of small video elements to display everyone), and if any of those values change, then it notifies the server. The server will then respond with an appropriate "simulcast" layer for the video's new dimensions, or if that video element is offscreen/not visible then it'll pause that track until it's visible again!

04/22/26:
- Found a nifty library today called [nuqs](https://github.com/47ng/nuqs) for type-safe parsing of search params via a react-style `useQueryState` hook. [CopDB](https://app.copdb.org) stores table filter options in the URL string, and we've been using our own handrolled utilities for parsing search params into their expected types -- which work _well enough_ right now, but they've just kinda been hacked together as needed. So I'm stoked to see if this library would be a good replacement! One thing that I like about our utilities that I'm hoping to replicate with nuqs (which I think `useQueryStates` may be able to), is that all available search params are defined by an interface which corresponds to the request params the API expects. So instead of working with individual values parsed from the URL string, we get a full "filter" object that's populated with the defined search params which we just pass as the request to the API.

04/10/26:
- Made it through the week! Had some fun at work today - our bulk-customer import job is humming along nicely, and we've got a more realistic dataset (~7 million rows) to play with locally. So now we were able to get a rough idea for the performance of our phone number lookup query. 
- As expected, our first pass with `ilike`ing the phone number results in a full table scan, which is too slow! But the legend that is my partner added a [GIN index](https://www.postgresql.org/docs/current/gin.html) to that column, and without changing any application code, local queries went from >8s to 5-10ms! We've still got a bunch of cleaning up to do and need to actually get the thing deployed and running, but that was a neat lil reminder about the efficacy of indexes!
